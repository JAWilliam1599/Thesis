

## 4.X Risk Scoring Model

### 4.X.1 Overview

In conventional DevSecOps pipelines, security tools such as static analysis and vulnerability scanners provide severity classifications (e.g., Critical, High, Medium, Low). While useful, these classifications alone are insufficient for prioritizing risks in complex environments, particularly in hybrid cloud architectures where deployment context significantly affects the impact of a vulnerability.

To address this limitation, this research proposes a **multi-factor risk scoring model** that extends beyond severity by incorporating additional dimensions, including exploitability, runtime exposure, and confidence level. The objective is to enable **context-aware prioritization of security findings**, thereby improving decision-making in automated deployment pipelines.

---

### 4.X.2 Risk Scoring Model Definition

The overall risk score is computed as a weighted aggregation of four factors:

[
\text{Risk Score} = W_s S + W_e E + W_x X + W_c C
]

Where:

* (S): Severity score derived from security analysis tools
* (E): Exploitability score representing the ease of exploitation
* (X): Exposure score reflecting runtime deployment context
* (C): Confidence score indicating the reliability of the finding
* (W_s, W_e, W_x, W_c): Weight coefficients such that
  [
  W_s + W_e + W_x + W_c = 1
  ]

The final score is normalized to a range of 0 to 100.

---

### 4.X.3 Severity Normalization

Security tools produce categorical severity levels, which are mapped to numerical values as shown in Table 4.X.

**Table 4.X: Severity Mapping**

| Severity Level | Numerical Value |
| -------------- | --------------- |
| Critical       | 10              |
| High           | 8               |
| Medium         | 5               |
| Low            | 2               |

The normalized severity score is calculated as:

[
S = \frac{\text{Severity Value}}{10}
]

This normalization ensures consistency across multiple tools with different scoring systems.

---

### 4.X.4 Exploitability Assessment

The exploitability factor evaluates how easily a vulnerability can be exploited in practice. It is derived from characteristics such as the availability of public exploits, required access level, and attack complexity.

**Table 4.X: Exploitability Scoring**

| Condition                        | Score |
| -------------------------------- | ----- |
| Public exploit available, remote | 1.0   |
| Remote exploitation possible     | 0.8   |
| Local or complex exploitation    | 0.5   |
| Difficult to exploit             | 0.2   |

This factor helps differentiate between theoretical vulnerabilities and those actively exploitable in real-world scenarios.

---

### 4.X.5 Exposure Assessment in Hybrid Cloud

Exposure represents the degree to which a vulnerable component is accessible within the deployment environment. This factor is particularly important in hybrid cloud systems, where applications may operate across public cloud and on-premises infrastructures with different trust boundaries.

**Table 4.X: Exposure Scoring**

| Deployment Context                    | Score |
| ------------------------------------- | ----- |
| Public-facing service (e.g., API)     | 1.0   |
| Internal service (private network)    | 0.6   |
| On-premises isolated system           | 0.3   |
| Sandbox or non-production environment | 0.1   |

This dimension enables the model to assign higher risk to vulnerabilities exposed to external attackers, even if their intrinsic severity is moderate.

---

### 4.X.6 Confidence Evaluation

The confidence score reflects the reliability of a detected issue and helps mitigate the impact of false positives generated by static analysis tools.

**Table 4.X: Confidence Scoring**

| Detection Context               | Score |
| ------------------------------- | ----- |
| Confirmed by multiple tools     | 1.0   |
| High-confidence rule match      | 0.8   |
| Single tool detection           | 0.5   |
| Heuristic or AI-based detection | 0.3   |

Incorporating confidence ensures that findings supported by multiple independent analyses are prioritized over uncertain detections.

---

### 4.X.7 Weight Configuration

The weighting coefficients determine the relative importance of each factor. In this study, the following configuration is used:

[
W_s = 0.35,\quad W_e = 0.25,\quad W_x = 0.25,\quad W_c = 0.15
]

This configuration reflects the following design principles:

* Severity remains a primary factor but does not dominate the model
* Exploitability and exposure are equally important in assessing real-world risk
* Confidence is used to adjust prioritization based on detection reliability

---

### 4.X.8 Risk Classification and Decision Policy

Based on the computed risk score, findings are classified into four levels, each associated with a specific action in the deployment pipeline.

**Table 4.X: Risk Levels and Actions**

| Score Range | Risk Level | Action                  |
| ----------- | ---------- | ----------------------- |
| 80–100      | Critical   | Block deployment        |
| 60–79       | High       | Require manual approval |
| 40–59       | Medium     | Allow with warning      |
| < 40        | Low        | Log for monitoring      |

This classification enables automated enforcement of security policies within the SysSecOps workflow.

---

### 4.X.9 Integration into SysSecOps Pipeline

The risk scoring model is integrated as a dedicated stage within the security pipeline, following static analysis and vulnerability scanning. The process is defined as follows:

1. Collect findings from security tools (SAST, dependency scanning, IaC analysis)
2. Normalize severity values across tools
3. Compute exploitability, exposure, and confidence scores
4. Calculate the overall risk score
5. Apply classification rules to determine the appropriate action

The output of this stage is a structured risk assessment for each finding, which is then used by the decision engine to enforce deployment policies.

---

### 4.X.10 Discussion

The proposed model provides several advantages:

* **Context-awareness:** Incorporates runtime deployment context, particularly important in hybrid cloud environments
* **Improved prioritization:** Distinguishes between high-severity but low-impact issues and moderate-severity but highly exposed vulnerabilities
* **False positive reduction:** Uses confidence scoring to reduce noise from automated tools
* **Automation readiness:** Produces quantitative outputs suitable for automated decision-making

However, the model also introduces additional computational overhead and depends on accurate estimation of exploitability and exposure factors.
